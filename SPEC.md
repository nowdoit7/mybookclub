## 1. Product overview

### One-liner
An AI-powered book club: finish a book, enter its title, and sit down at a round
table with 3 AI readers — each with a distinct interpretive lens, a lifetime
bookshelf, and opinions they actually defend.

### The problem
Fewer and fewer people have someone to talk to about the book they just finished.
Real book clubs are the single best way to deepen a reading — you discover that
the same book was read three completely different ways — but most readers don't
have access to one.

### Why this isn't "just a ChatGPT prompt"
A single chatbot converges: it agrees with you, summarizes both sides, plays the
friendly narrator. A real book club **collides**. This app is different because:

1. **Committed stances** — each persona pre-generates private reading notes and
   defends that position throughout the session.
2. **Orchestration in code** — a deterministic state machine (not the LLM) runs
   the 5-stage meeting, controls speaking order, and **forces disagreement**:
   after the user speaks, at least one persona must push back.
3. **A persistent artifact** — every session ends with an auto-generated,
   structured **meeting recap document** (the kind a real book club secretary
   would write), downloadable and shareable.
4. **A purpose-built UI** — round-table view, text-length-aware automatic
   pacing with a hard stop on user turns, active/next-speaker cues, and a stance map.

### Target user story (for demo + README)
The developer is a member of a real corporate book club in Korea. The demo
starts with a book selected by the evaluator and compares the resulting recap
with the kind of record a real club would keep — same reading experience,
different interpretations preserved rather than flattened.

## 3. MVP scope

### IN (must ship)
- Web app (React + Vite), desktop-first responsive
- Book title input → book verification step → session start
- Persona pool of 8; **constrained random draw of 3** (one per category)
- Private per-persona **reading notes** generated once at session start
- 5-stage session flow run by a **code state machine** (moderator)
- Moderator utterances generated by LLM, logic 100% in code
- Per-utterance stateless GPT-5.6 calls with JSON-schema output
- **Rebuttal enforcement**: ≥1 persona challenges the user's stated position in the discussion stage
- Text-length-aware **automatic pacing** by default, with pause, skip-now, and
  manual Next controls; every user turn is a hard stop
- Round-table UI (avatars and active/next-speaker highlight) + one persistent
  current-dialogue panel; the full transcript opens separately on demand
- User text input at designated turns
- Live **stance map** (per-topic position of each participant, -2..+2)
- Korean/English session language selection before the session starts; the
  selected language remains fixed until a new session begins
- Explicit Mock/Live generation selector. Mock is the safe default; Live is
  enabled only when `/api/health` confirms a server-side key and clearly states
  that the session uses API credits
- **Meeting recap document** auto-generated at session end (rendered + downloadable as Markdown)
- Session persistence in `localStorage` (refresh-safe)
- Reproducible seed mode: the seed controls persona selection and deterministic
  mock behavior only; it never chooses or pre-fills the user's book

### OUT (roadmap slide only — do not build)
- Voice input/output
- Moderator personality selection (UI may show a disabled dropdown labeled "coming soon")
- Long-term library / cross-book memory
- Multi-user rooms
- Database / accounts

## 4. Tech stack

| Layer | Choice | Notes |
|---|---|---|
| Frontend | React 18 + Vite + TypeScript | Single-page app |
| Styling | Tailwind CSS | Fast, consistent |
| Backend | Minimal Node/Express server | Serve the static build, proxy `/api/generate` to OpenAI, and enforce lightweight abuse controls at that boundary |
| Model | `gpt-5.6` via the Responses API | Strict structured output for every model call; use the JavaScript SDK's Zod parsing helper |
| State | React state + `localStorage` | No DB |
| Deploy | Firebase Hosting + Cloud Functions for Firebase v2 | Hosted judging URL; exact local run instructions remain the fallback |

**Env:** `OPENAI_API_KEY` in a gitignored `.env` for local development and in
Google Cloud Secret Manager for Firebase deployment. It never enters the client
bundle, browser storage, or repository.

The browser implements the same `GenerationClient` contract for both modes.
Live mode sends strict inputs to six same-origin server routes for book
identification, reading notes, discussion-focus extraction, utterances, user
stance, and recap generation.
The server validates every request before calling OpenAI and validates every
structured response before returning it.

**Deployment safety:** the proxy enforces a small request-body limit, an
allowlist of production origins, an in-memory per-IP rate limit, and a
per-session model-call ceiling. This transient operational state is not a
product session store and does not weaken the `localStorage`-only rule. Configure
a hard project usage limit in the OpenAI Platform before publishing the demo.

## 5. Architecture

```
┌─────────────────────────────────────────────┐
│ MODERATOR ENGINE (pure code, deterministic) │
│ · 5-stage state machine                     │
│ · speaking-order scheduler                  │
│ · rebuttal-enforcement trigger              │
│ · user-turn scheduler                       │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ PROMPT ASSEMBLER (per utterance)            │
│ persona card + private reading notes        │
│ + recent transcript window + directive      │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ GPT-5.6 (one stateless call per utterance)  │
│ JSON schema: {utterance, stance, refers_to} │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│ SHARED TRANSCRIPT + UI                      │
│ round table · chat log · stance map         │
└──────────────┬──────────────────────────────┘
               ▼
        ↻ back to moderator engine
```

**Key principles**
- **Shared transcript, isolated identities.** All personas see the same public
  transcript; each persona's card + reading notes are private context.
- **Orchestration is code, speech is model.** The LLM never decides what stage
  we're in or who talks next.
- **Prefetch.** While the user reads utterance N, generate N+1 in the background.
  Only prefetch when the next directive is fully determined by code. Invalidate
  the prefetch if the user changes the expected next directive before reveal.

### Token, call, and latency budget
- System (persona card + reading notes): ~1.2–1.8K tokens
- Transcript window (last ~12 utterances, older ones summarized to 1 line each): ~0.8K
- Directive: ~0.1K
- Output: ≤220 tokens
- Generate the three private reading-note objects in parallel.
- Run one discussion topic by default. A second topic is optional only when the
  session is within the latency/call budget.
- Expect roughly 30–40 model calls for a full live session; this is a latency
  risk even when token cost is modest. Track call count and elapsed time in the
  console harness.
- Use the golden transcript for UI development and deterministic replay only.
  The submitted demo path must still exercise GPT-5.6 for live generation.

## 7. Session flow — the 5-stage state machine

Global rules enforced by code:
- Persona utterances: **2–4 sentences**; moderator utterances: **1–3 sentences**.
  Prompt instructions and `maxLength` are guards, not proof. Count sentences in
  application code, perform one structured repair retry on failure, then use a
  short role-appropriate fallback and log the validation failure. Never truncate
  generated prose mid-sentence.
- A persona may cite its bookshelf **at most once per stage** (tracked in code; the
  directive says "you may/may not reference your shelf this turn").
- The user is prompted at fixed points; the app waits for input (no timeout in MVP).
- AI turns auto-advance after a language-aware reading delay derived from text
  length and sentence count, clamped to 3–10 seconds. The first turn may use a
  shorter entrance delay.
- Auto playback pauses when the page becomes hidden. The user can pause,
  continue, reveal the next turn immediately, or switch to manual mode; manual
  "Next" advances exactly one transition.
- Reaching a user turn always stops playback and exposes the input form. It
  never auto-passes or submits on the user's behalf.

### Stage 0 — Book identification and user confirmation (pre-session)
1. User chooses `single_book` or `series`, then enters a book/series title
   (+ optional author). The selected scope is binding for verification and every
   later generation call.
2. Mock/live mode is selected before identification. Mock mode returns an
   explicitly unverified, book-agnostic preview and never claims book knowledge.
3. In live mode, one GPT-5.6 Responses API call must use the hosted `web_search`
   tool with `tool_choice: required`. It returns a strict structured result with
   canonical title, author, selected scope, included component titles, 4-6
   sentence spoiler-tolerant summary, main characters, 3 candidate discussion
   topics, and a verification status. A verified single book has exactly one
   included title; a verified series has at least two titles in publication order.
4. Code extracts up to 3 HTTPS URLs from the API's actual web-search source
   metadata; the model never invents source URLs. `verified` requires the model's
   match plus at least two distinct retrieved sources. Otherwise downgrade to
   `ambiguous` or `not_found`.
5. Show verification status, scope, series component titles when applicable,
   note, and clearly visible clickable source links. URLs and citation markup are
   forbidden in the summary and topic fields; source links live only in the
   dedicated source area.
   Live `ambiguous`/`not_found` results cannot begin a session; the user must
   correct the title/author and search again. A verified result still requires
   the user's "Yes, that's the book" confirmation before becoming
   `confirmedSummary`.

**Copyright rule (bake into every prompt):** discuss themes, scenes, and
interpretations; quote at most a short phrase; never reproduce passages.

### Stage 1 — INTRO
- Moderator welcomes the user, briefly names the book, and lowers the social
  temperature: before discussing it, everyone will first say who they are.
- Each persona gives a social introduction in character: name, occupation/life
  context, and reading habit only. They do **not** offer an interpretation of the
  current book yet. Keep this to 2 short sentences.
- The UI keeps each person's short occupation/role visible beside their name so
  the user does not have to memorize three biographies.
- Moderator invites the user to share their name/background and what brought
  them to the table. The answer need not be about the book.
- **User turn** (free text, may skip).

### Stage 2 — FIRST_IMPRESSIONS
- Moderator: "Let's go around — first impressions?"
- Each persona: 1 independently prepared utterance expressing a clear overall
  take from its notes. Persona impressions are generated before any of them are
  revealed, so they do not mechanically echo the previous reader.
- **User turn**: user shares their impression.
- First impressions are personal testimony, not debate. No persona rebuts the
  user in this stage; the extracted stance is retained for later topic selection.

### Stage 3 — MEMORABLE_SCENES
- Moderator: "A scene or line that stuck with you?"
- Code selects two personas with meaningfully different overall positions to
  share one specific scene each. **At least one must surface an easily-missed
  scene** from its `keyScenes` ("Did anyone else catch…").
- **User turn**: user shares theirs. One of those readers reacts once. The third
  persona may remain silent; being seated does not create a speaking obligation.

### Stage 4 — DISCUSSION (the main event)
- Code selects **1 topic by default** from the confirmed book's candidate topic
  list or one grounded emergent question. It combines structured relevance
  scores from the real Stage 2/3 conversation with persona stance spread. The
  moderator explains the conversational thread that led to the selected question
  and states it crisply. A second topic is optional only when the live session remains within its
  latency/call budget.
- Round structure per topic (code-controlled):
  1. Moderator names the user-grounded controversy and asks the user to state a position.
  2. User answers → stance extraction (§8).
  3. The persona furthest from the user becomes the **primary challenger**,
     directly rebuts the claim, and asks one pointed question (§9).
  4. **The next turn must belong to the user.** The challenge is not complete
     until the user can answer it; extract and update the user's stance again.
  5. The same challenger responds once to that answer without asking another
     question or resetting the topic.
  6. The persona closest to the user's updated stance becomes the **supporter**
     and contributes once: strengthen the user's case with different evidence
     while naming one limit. The third persona is the **observer** and does not
     speak merely to complete a round.
  7. Moderator closes with a one-line tension summary, never a resolution.
- This creates a spotlight conflict: **user ↔ challenger**, with one supporting
  intervention. Max ~8 generated/user utterances per topic.

### Stage 5 — WRAP_UP
- Moderator asks the user first for a closing thought ("Did this discussion move you?").
- **User turn** comes before AI closings so a new final idea is not ignored.
- The discussion's challenger and supporter each give one short response to the
  user's closing thought. The observer is not forced to speak.
- Moderator gives a substantive 2–3 sentence spoken summary: central
  disagreement, what shifted, and what remains unresolved. After the summary's
  normal reading delay, trigger **recap generation** (§10) → recap screen.

## 8. User input handling

For every user utterance:
1. Append to transcript as `speaker: "user"`.
2. Blank input means pass; do not call the model.
3. Run a **stance extraction call** only in:
   - `FIRST_IMPRESSIONS`, using the synthetic target `overall_impression` and
     each persona's `overallStance`; and
   - `DISCUSSION`, using the current `activeTopic`.
4. Skip stance extraction in `INTRO`, `MEMORABLE_SCENES`, and `WRAP_UP`.
5. Store the -2..+2 stance plus 1-line paraphrase. The moderator engine uses it
   for rebuttal targeting and the stance map.

User input UI: single-line-growing textarea, appears only on user turns,
placeholder varies by stage ("Your first impression…").

## 9. Rebuttal enforcement (anti-sycophancy) — core differentiator

When the user states a position in DISCUSSION,
resolve `personaStance` from `stanceByTopic[activeTopic]` or from
`overallStance` only for ranking context; Stage 2 never triggers a rebuttal:

```
challenger = persona whose notes.stanceByTopic[activeTopic]
             differs most from userStance
             (tie → random among most-distant; never the same persona twice in a row)

directive to challenger:
  "The user just argued: {paraphrase}. Your private notes disagree
   ({stance}, because {reason}). Respectfully but firmly challenge them.
   Ask them one pointed question. Do NOT concede. 2-4 sentences."
```

The pointed question opens a second user turn immediately. After the user
answers, the challenger responds once and the supporter contributes once. No
other persona exchange is scheduled for symmetry.

If `|userStance - all persona stances| < 0.5` (everyone genuinely agrees),
the moderator itself plays devil's advocate: "Let me push back for a moment…"
— so the guarantee holds: **the user is always challenged at least once.**

Personas may be *persuaded* over a session, but only explicitly:
"I'll concede X" — silent drift is forbidden in their cards.

## 10. Meeting recap document (session artifact)

Generated by one GPT-5.6 call over the full transcript at WRAP_UP as strict
`{ markdown }` JSON, rendered
in-app and downloadable as `recap-{book}-{date}.md`.

Template (mirrors a real book-club recap):

```md
# {Book Title} — Reading Table Recap, {date}

## Discussion summary
(3-5 sentences)

## Where everyone landed
| Topic | {P1} | {P2} | {P3} | You |
|---|---|---|---|---|
| {topic 1} | {stance + 1-line} | … | … | … |

## Sparks — moments of real disagreement
- {at most 2 short excerpts of the best app-generated exchanges, incl. the user's}

## Scenes you might have missed
- {at most 3 items from personas' keyScenes that surfaced}

## From the shelves
- {bookshelf citations made during the session, with why they fit}

## A question to sleep on
{exactly one open question the moderator leaves the user with}
```

The stance map final state is embedded as the table above. This document is the
demo's closing shot: real club recap (left) vs. app recap (right).

Recap excerpts may quote only the session transcript, not passages from the
book. Apply the same short-phrase copyright rule during recap generation and
validation. Do not include private reading notes verbatim in the downloaded
artifact.

```
│  CURRENT DIALOGUE (full active utterance)       │
│  [ View conversation transcript ]              │
├────────────────────────────────────────────────┤
│  [ stance map ] [pause/next now/manual control]│
│  (user textarea appears here on user turns)    │
└────────────────────────────────────────────────┘
```

- Avatars: colored initial chips (no image assets in MVP). Distinct color per persona.
- Book entry includes a Korean/English selector. The selected language controls
  UI copy, generated dialogue, and recap headings, and cannot change mid-session.
- Active speaker: chip scales up and a dedicated dialogue panel shows the full
  current utterance. The panel is the session's primary reading surface and
  reserves a portrait slot that can later use character art without changing
  the session engine.
- The moderator, three personas, and the user sit around the table. Highlight
  only the current and next speaker; do not expose the full future queue.
- Do not render the growing transcript beneath the table. Append every turn to
  session state silently and expose the full speaker-tagged transcript through
  a separate modal or drawer. Opening it pauses automatic playback.
- Automatic pacing is the default. Delay is calculated from the visible text,
  with pause/continue and "Next now" controls plus a manual-mode toggle. Controls
  are disabled while generation is in flight. Timing stays invisible: do not
  show seconds or a progress bar. Page visibility loss pauses the pending
  transition so dialogue is not skipped in a background tab.
- User turns are visually distinguished at the table and in the input area and are a
  hard stop regardless of the selected pacing mode.
- Stance map: slide-up panel; horizontal axis per topic (-2 critical … +2 sympathetic),
  dots = participants, animated when stances update.
- Recap screen: tabs for rendered meeting recap and the complete conversation,
  plus recap copy, "Download .md", transcript copy, and "Start a new session".
- A "Copy full transcript" action is available inside the session transcript
  view and on the recap screen. It copies evaluation-ready Markdown with stage
  headings, speaker names, and utterance text, excluding decorative avatar initials.
- Before the first model call, show this compact disclosure near the start button:
  "Your book title and messages are sent to OpenAI to generate this discussion.
  This session is stored only in this browser and is not saved by The Reading
  Table's server."
- "Start a new session" clears the active session data from `localStorage`
  before returning to book entry.

Tone: warm library feel — cream/paper background, serif for book title,
sans for chat. Keep it simple; polish > features.

### 11.1 Data & privacy

- Book details, user messages, and the minimum transcript context needed for a
  turn are sent through the server proxy to OpenAI for generation.
- Product session state is stored only in the current browser's `localStorage`.
- The application server has no transcript database and no server-side product
  session store. Do not log request bodies, prompts, reading notes, or transcripts.
- Infrastructure providers may retain operational request metadata under their
  own policies; do not claim that no infrastructure logs exist.
- Provide an explicit local "Start a new session" deletion path.
- `README.md` must contain a Data & Privacy section describing data flow,
  browser storage, OpenAI processing, server non-persistence, and deletion.
- Firebase Auth, Firestore history, and cross-device sync remain roadmap items,
  not hidden MVP dependencies.

## 12. Persona system

### 12.1 Draw rules
- Pool of 8, categories: emotional ×3, analytical ×3, contextual ×2.
- Draw **one persona per category** in normal sessions (guarantees productive friction).
- A supplied seed may reproduce a persona trio for testing or a demo video, but
  the book always comes from Stage 0 user input. No seed may inject book facts,
  characters, scenes, topics, or scripted user opinions.

### 12.1.1 Mock-mode truthfulness
- Mock mode must accept any title so the entire session flow can be tested
  without credits, but it must not pretend to know that book's plot or context.
- Mock output may use the entered title, author, persona data, candidate
  questions, and statements already present in the transcript. It must not
  invent book-specific people, scenes, quotations, or historical claims.
- Live mode is responsible for model-identified, book-specific content. The UI
  labels mock identification as a structural preview rather than verification.

### 12.2 System prompt assembly (per utterance)
```
[PERSONA CARD]           ← static, from card
[YOUR PRIVATE READING NOTES ON "{book}"]  ← this session's notes
[HOUSE RULES]            ← global rules (length, copyright, no narrator-mode)
[RECENT CONVERSATION]    ← last ~12 utterances (older: 1-line summaries)
[DIRECTIVE]              ← from moderator engine, e.g.:
   "Stage: DISCUSSION. Topic: '{topic}'. Respond to {speaker}.
    Shelf reference: not allowed this turn. Respond with JSON only."
```

### 12.3 House rules (shared by all personas)
```
- Speak as your character. 2-4 sentences. No monologues.
- On social introductions, say what you do and how you tend to read; save all
  opinions about the current book for FIRST_IMPRESSIONS.
- Never act as a neutral narrator or summarize "both sides."
- Never invent plot details. If unsure about a scene, hedge in character
  ("if I'm remembering right…").
- Discuss themes and scenes; quote at most a short phrase; never reproduce
  passages from the book.
- Disagree when your notes disagree. Concede only explicitly.
- Address others by name. React to what was actually said.
- Let earlier remarks put pressure on your thinking. Refer to a precise phrase,
  admit uncertainty when it is real, and avoid polishing every turn into a conclusion.
- Avoid repetitive debate templates such as "both can coexist," "not A but B,"
  or automatic praise followed by "however."
- Reply with the JSON object only.
```

### 12.4 Reading-notes generation prompt (once per persona per session)
```
You are {name}. {identity} {lens}
Your formative bookshelf: {bookshelf with takeaways}

The club is discussing "{title}" by {author}.
Book context: {confirmedSummary}
Candidate topics: {topics}

Produce your PRIVATE reading notes as JSON:
- overallTake: your honest, committed 2-3 sentence position, in your voice
- overallStance: your overall evaluation from -2 to +2
- stanceByTopic: for each candidate topic, a stance from -2 to +2 and a reason
  consistent with your lens
- keyScenes: 2-3 specific scenes you'd cite — include at least one scene a
  casual reader easily misses
- shelfConnections: 1-2 genuine links between this book and your bookshelf
  (only if truly illuminating; otherwise fewer)
- personalReaction: the scene or idea that genuinely moved, irritated, or puzzled you
- unresolvedQuestion: a doubt you cannot neatly settle before the meeting
- possibleRevision: what evidence or argument could genuinely move your position
- questionForTable: one question you actually want to ask another reader
Stay in character. These notes are your anchor for the whole session.
```

Generate these private notes with GPT-5.6 at medium reasoning effort. Keep live
utterance calls latency-sensitive at low effort. The three note calls remain
parallel and persona introductions are prefetched as each note becomes ready.

## 15. Error handling & edge cases

- **API failure mid-session:** retry ×2 with backoff → toast "The table lost its
  train of thought — press Next to retry." State machine can always regenerate the
  current utterance idempotently.
- **Schema or sentence-count failure:** one structured repair retry with the
  exact validation error; then a role-appropriate fallback utterance and log.
  Never append invalid output to the transcript.
- **Model refusal:** detect the Responses API refusal item explicitly, show a
  safe stage-appropriate message, and preserve the pending engine turn for retry.
- **User writes something off-topic/blank:** moderator gently redirects once;
  blank = "pass," moderator moves on without stance extraction.
- **Obscure or ambiguous book:** show the retrieved evidence and keep the live
  start action disabled until title and author are verified; never silently fall
  back to model memory or invent scenes.
- **User tries to jailbreak personas:** house rules keep them in character;
  moderator redirects to the book.
- **Rate limit reached:** return a typed 429 response with a retry delay; preserve
  the pending engine turn so the user can retry without duplicating transcript state.
- **Session call ceiling reached:** stop generation with a clear message rather
  than falling through to paid, unbounded requests.

### Deployment readiness

- Publish the React build on Firebase Hosting and expose the OpenAI proxy as a
  Cloud Functions for Firebase v2 HTTPS function. Use a Hosting rewrite so the
  browser calls the same-origin `/api/generate` path.
- Bind `OPENAI_API_KEY` from Google Cloud Secret Manager to the function; never
  deploy the local `.env` file.
- Publish a working Firebase demo URL for judging; local execution is the documented fallback.
- Restrict production CORS to the deployed frontend origin.
- Reject oversized or malformed bodies before any OpenAI call.
- Use transient, in-memory per-IP throttling and a client session id for a
  best-effort per-session call ceiling. These are abuse guards, not authentication.
- Set a hard OpenAI project usage limit before the URL is shared.
- Provide a reproducible seed, `.env.example`, sample data where needed, and exact test
  instructions plus Data & Privacy in `README.md`.
